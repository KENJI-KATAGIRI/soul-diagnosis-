# 導線設計：外部（本システム）→ UTAGE

## 原則

| レイヤー | 役割 |
|----------|------|
| **外部LP・診断・結果** | 感情・理解・共感・パーソナライズ体験 |
| **UTAGE** | 登録・タグ付け・ステップ配信・ウェビナー・決済・会員 |

UTAGEは「補助」ではなく、**回収・最適化・配信のエンジン**。

---

## 推奨フロー（ゴール構成との対応）

```
① 外部LP（診断付き）     … 本アプリ `/` または独自LPから `/` へ
② 診断AI                 … 任意：`/shindan-ai/`（別ルート）
③ 結果ページ             … `POST /result` → 結果画面
④ メール登録（外部）     … 結果画面の登録フォーム、またはUTAGE単体LP
⑤ UTAGEへPOST登録        … 本アプリ `POST /lead/utage`（サーバからWebhookへ）
⑥ 以降                   … UTAGE（ステップ・ウェビナー・講座・会員）
```

---

## 分岐パターン

### A. メイン（推奨）

1. ユーザーが**10問診断**を完了  
2. **結果ページ**で共感・問題・未来を読む  
3. 同じ画面で**メール登録**（任意）  
4. サーバが**診断サマリ付きでUTAGE**へPOST  
5. UTAGEが**タグ・シナリオ**を付与し、メール登録完了ページ／ステップ配信へ  

**メリット**: 診断データが揃った状態でUTAGEに届く。

---

### B. サブLPのみ（広告・A/B用）

1. UTAGEの**ショートLP**でメールだけ取得  
2. メール内リンクで**本アプリの診断**へ誘導  
3. 診断後、再度UTAGEに**タイプ情報を送る**（別Webhookまたは更新APIが必要な場合はUTAGE側仕様に合わせる）  

**本実装の標準**は **A（結果ページからPOST）** です。

---

### C. 診断なしでUTAGE登録だけ

- UTAGE単体で完結  
- 本システムとは**データ連携なし**（または手動CSV）  

---

## CTAの役割分担

| 場所 | CTA例 | 行き先 |
|------|--------|--------|
| 結果ページ上部〜中央 | 本格レポート／次のステップ | 本アプリ内 |
| 結果ページ **登録ブロック** | メール登録 | `POST /lead/utage` → UTAGE |
| フッター講座 | 講座申込 | `SOUL_COURSE_PURCHASE_URL`（外部） |

**コンバージョン最大化のポイント**: 登録は**結果を読んだ直後**に1ブロック置く（スクロール内の自然な位置）。
